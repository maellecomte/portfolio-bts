Affichage
dynamique v2

Technologies utilisées

Digital signage : API NestJS, SPA React 18 (TypeScript, Vite, Tailwind), PostgreSQL 16 (TypeORM), fichiers sur MinIO (S3), temps réel Socket.IO, Docker Compose + Nginx.

Cliquer pour le détail

Nature du projet

Application de digital signage : une API métier NestJS et une SPA React 18 (TypeScript, Vite, Tailwind), servie derrière Nginx, avec persistance PostgreSQL 16 via TypeORM et stockage des médias binaires sur MinIO (API compatible S3, pilote aws-sdk côté backend).

Déploiement

Docker Compose orchestre en général : base de données, MinIO, backend, frontend (build statique + Nginx dans l’image ou selon Dockerfile), reverse proxy Nginx exposé (souvent port 80). Selon la branche, d’autres services peuvent figurer dans le compose (ex. brique réseau type pfSense).

Backend (NestJS)

Architecture modulaire (src/modules/*) : authentification JWT / Passport, médias (upload Multer, stockage piloté par STORAGE_DRIVER: s3), playlists et items, canaux (écrans), planning, analytics, audit, device tokens, health, etc.

  • Temps réel : @nestjs/websockets + Socket.IO pour pousser les mises à jour vers les clients.
  • Sécurité : compte admin unique par instance ; routes publiques orientées player (infos canal, items de playlist, rendu média, validation de token, heartbeat) face aux routes protégées JWT pour l’interface d’administration.

Frontend (React)

Pages métier : dashboard des écrans, médiathèque, playlists, planning, analytics, paramètres, PlayerView en mode kiosk, éditeur d’écran, médias dynamiques. État avec Zustand, requêtes HTTP avec Axios (lib/api.ts), WebSocket dédié aux canaux. Service Worker pour stratégie de cache / usage offline (selon la doc du dépôt).

Flux de données clés

  1. Admin authentifié — CRUD médias, playlists, canaux, planning via les routes /api.
  2. Player (non authentifié ou via token) — URLs publiques, chargement de la playlist, affichage des médias (dont PDF via react-pdf / pdf.js).
  3. Heartbeat HTTP périodique depuis le player pour marquer un canal en ligne ou hors ligne côté serveur.

Configuration sensible

Backend : DSN Postgres, JWT_SECRET, endpoint et identifiants S3/MinIO, PUBLIC_MEDIA_URL et WS_PUBLIC_URL (doivent refléter l’URL telle que vue par le navigateur). Frontend buildé avec variables Vite : VITE_API_URL, VITE_WS_URL, VITE_BACKEND_URL — une mauvaise IP ou un mauvais schéma ws/wss casse médias ou WebSocket.

Intégrations optionnelles

Clés pour OpenAI, Hugging Face, Unsplash, OpenWeatherMap peuvent être injectées côté backend pour widgets IA, images ou météo, selon ce qui est réellement branché dans les modules concernés.

En résumé

Stack SPA + API REST + WebSocket + Postgres + object storage, packagée pour un déploiement on-premise, orientée écrans physiques en kiosk avec suivi de présence par heartbeat.

03

Application

Badgeuse
chauffeurs

Technologies utilisées

Python, Raspberry Pi, SQLite, CustomTkinter, sync HTTP vers API (POST /sync), badge iButton ou saisie clavier, configuration centralisée (config.py).

Cliquer pour le détail

Objectif métier

Application de pointage des chauffeurs sur site : enregistrement local des passages ENTRÉE / SORTIE, puis remontée vers une API distante pour le suivi centralisé (multi-sites possible côté serveur).

Flux utilisateur

  1. Lecture du badge (iButton ou identifiant au clavier).
  2. Choix de l’action (entrée / sortie) et contexte jour si prévu (groupe, entreprise, site).
  3. Enregistrement immédiat en SQLite avec horodatage local.
  4. Synchronisation en arrière-plan : POST vers /sync ou /api/sync avec authentification (ex. X-API-KEY), gestion des erreurs et retry.
  5. Retour visuel pour l’opérateur : succès, erreur réseau ou données refusées.

Architecture logicielle

Découpage modulaire type :

  • config.py — paramètres : SITE_ID, WEBHOOK_URL, API_KEY, DB_PATH, timeouts.
  • main.py — point d’entrée : initialisation DB, lancement UI et boucle de sync.
  • badgeuse_ui.py — interface CustomTkinter mode kiosk : grande horloge, écrans d’état, digicode, changement de site si prévu.
  • hardware_manager.py — abstraction lecteur / clavier, normalisation de l’identifiant badge.
  • database_manager.py — insertions, lecture des pointages non synchronisés, mise à jour du statut.
  • sync_manager.py — client HTTP, construction du JSON, codes de retour, stratégie de nouvelle tentative.
  • session_manager.py — contexte « journée » : groupe, entreprise, site courant alimentant le payload API.

Données envoyées à l’API (exemple)

Payload typique : id_badge, site_source, action, timestamp_local, groupe_du_jour, entreprise_du_jour. Normalisation métier : ENTRÉE → ARRIVÉE, SORTIE → DÉPART ; identifiant badge nettoyé (casse, espaces) selon le contrat serveur.

Réseau & robustesse

  • Détection online / offline pour adapter l’UI et la fréquence de sync.
  • Boucle de synchronisation : traitement des enregistrements en attente par petits lots.
  • Réponses HTTP : succès (2xx) → marquage synchronisé ; 400 / 404 → erreur de requête (souvent sans retry identique) ; 401 / 403 → clé ou droits ; 429 → respect du débit (backoff) ; 5xx / timeout → nouvelles tentatives puis file d’attente conservée en local.

Base SQLite

Table des pointages avec indicateur is_synced : 0 = en attente d’envoi ou échec à traiter, 1 = accusé réception serveur. Requêtes clés : insertion après scan, sélection des lignes non sync ordonnées, mise à jour après POST réussi ou journalisation de la dernière erreur.

Interface opérateur

Machine d’états du type IDLE → SCANNED → SUCCESS ou ERROR : feedback lisible, timeouts de retour à l’accueil, gestion des badges inconnus selon la règle métier (refus ou enregistrement signalé).

Exploitation

En cas de « ça ne part plus » : vérifier WEBHOOK_URL, la clé API, les logs sync_manager, et une requête de test curl en POST JSON depuis le même réseau ; contrôler en base le nombre de lignes avec is_synced = 0.
